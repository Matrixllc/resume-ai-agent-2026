# 产品说明：简历智能问答系统

## 1. 产品定位

本系统定位为 **招聘简历智能检索与证据问答系统**。

它不是让 AI 直接替招聘人员决定“录用谁”或“淘汰谁”，而是帮助招聘人员更快完成这些工作：

- 查询候选人数量、列表和基本信息。
- 按技能、领域、项目经验筛选候选人。
- 解释某个候选人的经验体现在哪里。
- 对比两个候选人的简历事实和项目证据。
- 基于 JD 对候选人做初筛排序。
- 给每个回答提供可追溯的工具结果和证据来源。

一句话概括：

> LLM 负责理解问题和组织表达，事实、数量、候选人 ID、证据、排序分数必须来自工具。

## 2. 整体 AI 链路

系统采用受控编排链路，而不是自由 Agent。

```text
用户问题
-> router
-> planner
-> plan_validator
-> executor
-> execution_validator
-> aggregator
-> answer_validator
-> 最终答案
```

各节点职责如下：

| 节点 | 职责 | 不允许做的事 |
|---|---|---|
| router | 判断问题类型、是否复合问题、是否需要 JD 或证据 | 不查库、不回答 |
| planner | 生成结构化工具计划 | 不执行工具、不编造事实 |
| plan_validator | 校验工具、参数、依赖、边界 | 不修答案、不查数据 |
| executor | 唯一执行工具的节点 | 不总结、不排序重算 |
| execution_validator | 校验工具结果是否足够回答 | 不生成最终答案 |
| aggregator | 基于工具结果生成中文答案 | 不新增工具结果外的事实 |
| answer_validator | 校验答案是否越界 | 不改变工具结果 |

这种设计的目标是让 AI 问答可控、可审计、可追溯。

## 3. 支持的问答场景

### 3.1 候选人数量

示例：

```text
有多少个金融领域候选人？
```

典型链路：

```text
router -> candidate_count
planner -> filter_candidates -> count_candidates
executor -> 先筛选，再计数
aggregator -> 返回数量
```

适合回答：

- 一共有多少个候选人？
- 有多少个 Python 候选人？
- 有多少个金融领域候选人？

### 3.2 候选人列表

示例：

```text
推荐系统经验的候选人都有谁？
```

典型链路：

```text
router -> candidate_list / candidate_filter
planner -> filter_candidates 或 hybrid_search_candidates
executor -> 查询候选人和命中原因
aggregator -> 返回候选人列表
```

适合回答：

- 都有谁？
- 有哪些做过推荐系统？
- 谁有 Python 和金融项目经验？

### 3.3 个人信息介绍

示例：

```text
孟连星个人信息
```

典型链路：

```text
resolve_candidate_reference
-> get_candidate_profile_intro
```

回答内容包括：

- 姓名
- 求职意向
- 地区
- 教育经历
- 工作经历
- 技能标签
- 项目概览

联系方式默认隐藏，除非用户明确询问联系方式。

### 3.4 证据追问

示例：

```text
孟连星的推荐系统经验体现在哪？
```

典型链路：

```text
resolve_candidate_reference
-> search_candidate_evidence
```

系统会从项目级证据中总结关键依据，而不是直接堆长原文。

示例回答形态：

```text
主要依据：
- 孟连星在电影推荐系统开发-机器学习中体现：使用 Python 网络爬虫框架收集电影和用户信息。
```

### 3.5 双人比较

示例：

```text
孟连星和孔德程谁更好？
```

典型链路：

```text
resolve_candidate_reference
-> build_comparison_pack
```

回答边界：

- 只能比较两个人。
- 不引入第三个候选人。
- 没有 JD 时不能做绝对化判断。
- 可以基于当前简历事实和证据给出有边界的倾向。

示例表达：

```text
没有岗位 JD 时不能判断绝对“谁更好”。
按当前简历事实和证据包，如果更看重项目数量和可用证据，更倾向孔德程；
如果更看重全栈开发和后端方向，孟连星也有匹配点。
```

### 3.6 JD 排序与初筛

示例：

```text
按默认 JD 给候选人排序
```

典型链路：

```text
load_default_jd_criteria
-> filter_candidates
-> score_candidates_for_jd
-> rank_candidates
```

排序规则：

- 排序顺序只能来自 `rank_candidates`。
- aggregator 不能自行调整顺序。
- 用户没提供 JD 时，使用默认 JD，并在答案中说明。
- 排序属于辅助初筛，不代表最终招聘决策。

### 3.7 复合问题

示例：

```text
有多少个候选人，都有谁，按默认 JD 排序
```

系统会识别为复合问题，并拆成多个子任务：

```text
candidate_count
candidate_list
candidate_ranking
```

每个子任务独立规划工具链，最终聚合成一个答案。

## 4. 工程实现说明

### 4.1 Deterministic Planner

部分场景的工具链非常固定，例如：

```text
candidate_profile_intro
candidate_compare_pair
candidate_ranking
```

这些场景会使用 deterministic planner，也就是确定性计划生成。

例如“孟连星个人信息”的计划固定为：

```json
[
  {
    "name": "resolve_candidate_reference",
    "arguments": {
      "text": "孟连星个人信息"
    },
    "output_key": "resolved_candidate"
  },
  {
    "name": "get_candidate_profile_intro",
    "arguments": {
      "resume_identity": "$resolved_candidate.candidate_ids.0"
    },
    "depends_on": ["resolved_candidate"]
  }
]
```

这里 planner 没有提前拿真实候选人 ID，而是使用引用：

```text
$resolved_candidate.candidate_ids.0
```

真实 ID 只会在 executor 执行第一步工具后绑定。

这样做的好处是：

- 避免 LLM 编造 candidate_id。
- 避免 LLM 漏掉依赖。
- 降低成本和延迟。
- 固定场景结果更稳定。
- 仍然保留 plan_validator 校验。

### 4.2 工具依赖引用

工具之间的数据依赖通过 `output_key` 和 `$变量路径` 表达。

示例：

```text
resolve_candidate_reference(output_key="resolved_candidate")
get_candidate_profile_intro(resume_identity="$resolved_candidate.candidate_ids.0")
```

executor 会维护本轮 `tool_context`：

```text
tool_context["resolved_candidate"] = 第一步工具结果
```

然后在执行第二步前解析：

```text
$resolved_candidate.candidate_ids.0
```

如果引用解析失败，executor 返回失败的工具结果，不会静默猜测。

### 4.3 只读工具

问答系统调用的工具全部以只读为主：

- 查询候选人列表
- 筛选候选人
- 解析候选人引用
- 获取候选人简介
- 查询候选人证据
- 构建比较包
- 加载 JD criteria
- 对候选人评分
- 排序候选人

问答层不负责简历入库，不写数据库，不修改 Chroma。

### 4.4 证据约束

系统把证据分为不同强度：

- 项目原文证据：强证据
- 项目标签 / 领域标签：中等证据
- 候选人标签：弱证据

证据类问题必须返回 evidence refs。  
如果没有证据，不能硬说候选人没有相关经验，只能说明当前入库数据未找到证据。

前端右侧默认只展示“本轮关键依据”，最多展示 3 条摘要后的证据。

### 4.5 Trace 审计

每次问答都会记录 trace，包括：

- router 判断结果
- planner 生成的工具计划
- plan validator 是否通过
- executor 调用的工具
- execution validator 是否通过
- aggregator 使用的是 LLM、deterministic 还是 fallback
- answer validator 是否通过
- answer preview 是否被截断

这让系统具备调试、复盘和客户解释能力。

## 5. 示例流转

### 示例 1：个人信息

用户问题：

```text
孟连星个人信息
```

链路：

```text
router: candidate_profile_intro
planner: deterministic
tools:
  1. resolve_candidate_reference
  2. get_candidate_profile_intro
aggregator: 生成候选人简介
answer_validator: 校验联系方式默认隐藏
```

输出重点：

- 孟连星是谁
- 求职意向
- 教育经历
- 工作经历
- 技能和项目概览

### 示例 2：经验证据

用户问题：

```text
孟连星的推荐系统经验体现在哪？
```

链路：

```text
router: evidence_question
planner:
  1. resolve_candidate_reference
  2. search_candidate_evidence
executor: 查项目证据
aggregator: 总结关键依据
```

输出重点：

- 哪个项目体现该经验
- 做了什么事情
- 使用了什么技术或承担了什么职责

### 示例 3：双人比较

用户问题：

```text
孟连星和孔德程谁更好？
```

链路：

```text
router: candidate_compare_pair
planner: deterministic
tools:
  1. resolve_candidate_reference
  2. build_comparison_pack
aggregator: 生成有边界的比较结论
```

输出重点：

- 没有 JD 时不能绝对判断谁更好。
- 按当前简历事实和证据给出倾向。
- 展示 2-3 条关键依据。

### 示例 4：JD 排序

用户问题：

```text
按默认 JD 给候选人排序
```

链路：

```text
router: candidate_ranking
planner: deterministic
tools:
  1. load_default_jd_criteria
  2. filter_candidates
  3. score_candidates_for_jd
  4. rank_candidates
aggregator: 输出排序结果和依据
```

输出重点：

- 排名
- 总分
- 亮点
- 风险
- 主要依据

## 6. 商业化可行性

这个系统具备商业化潜力，但产品定位要准确。

不适合包装成：

```text
AI 自动决定录用谁
```

更适合包装成：

```text
招聘简历智能检索与证据问答助手
```

适合的商业场景包括：

- 企业招聘团队内部简历检索。
- 猎头团队候选人知识库问答。
- ATS 系统的智能问答插件。
- JD 初筛和候选人排序辅助。
- 面试前候选人摘要和证据准备。

核心卖点：

- 减少人工读简历时间。
- 快速定位候选人经验。
- 对比较和排序给出可追溯依据。
- 避免 LLM 编造候选人事实。
- 支持审计和复盘。

## 7. 当前不足与产品化方向

### 7.1 证据质量还需要继续打磨

后续可以增强：

- 证据和结论逐条绑定。
- 强证据、弱证据前端分层展示。
- 空证据场景更谨慎地回答。
- 证据摘要更自然、更贴近招聘语言。

### 7.2 JD Scoring 需要产品化

后续可以增强：

- 支持上传多个 JD。
- 支持 JD 维度配置。
- 支持评分权重调整。
- 每个评分维度绑定证据。
- 输出可导出的候选人评估报告。

### 7.3 多轮上下文需要更稳

后续可以增强：

- “他 / 第一名 / 刚才那个人”的引用解析。
- 上下文过期策略。
- 多候选人歧义澄清。
- 会话级审计。

### 7.4 权限和隐私要补齐

商业化必须考虑：

- 字段级权限控制。
- 联系方式脱敏。
- 用户角色权限。
- 企业租户隔离。
- 数据删除和导出。
- 操作日志和审计。

### 7.5 评测集需要系统化

后续应建设标准回归集：

- 数量类问题
- 列表类问题
- 个人信息类问题
- 证据类问题
- 双人比较问题
- JD 排序问题
- 复合问题
- 歧义和空结果问题

每次修改 prompt、tool、validator、aggregator 后，都应跑回归验证。

## 8. 总结

这个系统的核心不是让 LLM 自由发挥，而是把 LLM 放进一个可控工程链路中：

```text
LLM 理解问题
工具提供事实
Validator 约束边界
Aggregator 组织表达
Trace 保证可审计
```

它适合做招聘场景里的智能检索、证据问答和 JD 初筛辅助。

商业化方向成立，但应该坚持辅助决策定位：

> AI 帮招聘人员更快读简历、找证据、做初筛；最终招聘判断仍由人负责。
